# Project Summary

Weather forecast dashboard MVP using React, TypeScript, and Tailwind CSS. The application integrates with Open-Meteo's Geocoding and Weather Forecast APIs to let users search for any city globally.

The app features a main Forecast View displaying real-time conditions (temperature, feels-like, wind, humidity, pressure, and custom weather icons mapped from WMO codes), a horizontal scrollable Hourly Forecast, and a paginated 30-Day Forecast (chunked into clean 5-day intervals).

Weather forecast application using React with TypeScript, created from scratch with Vite and styled with Tailwind CSS. The application connects to the Open-Meteo public API to search for cities in real time and display detailed 30-day forecasts and hourly information. The system features a search engine with optimized (debounced) autocomplete, visualization of current key metrics (temperature, wind chill, humidity, pressure, and sky conditions mapped from WMO codes), an hourly carousel, a paginated 30-day list. It also provides a favorites panel that displays information, that persists starred cities to localStorage, rendering them in a customized responsive table featuring dynamic country flag icons.

# Project Architecture

The project is structured modularly to keep the responsibilities of each file well separated:

  src/services/api.ts: Centralizes all asynchronous calls to the Open-Meteo API to return structured data in JSON format (cities and weather data).
  
  src/context/WeatherContext.tsx: Wraps the entire application in a global provider that manages the active city (activeCity), the array of favorite cities (favorites), and the methods for updating them, synchronously checking if a city is in favorites.
  
  src/hooks/useFetchWeather.ts: A custom hook that fires automatically whenever the activeCity context changes. It handles fetching the weather for that location and managing loading (isLoading) and error states.
  
  src/utils/weatherCodes.ts: The mapper that translates the WMO numeric codes returned by the API into descriptive Lucide React text and icons.
  
  src/components/: Pure and reusable UI components (SearchBar, WeatherHero, HourlyForecast, DailyForecast, FavoritesView).
  
  This structure has allowed me to completely isolate the network logic and global state from the visual components, making the code clean, readable, and easy to scale.

# Technical Decisions

Navigation and Views (currentView): Instead of using a heavyweight routing library, I set up a simple state in App.tsx called currentView. The buttons in the side menu (PC) or bottom menu (mobile) set this state to dynamically render the home screen or favorites view in the main viewport.

If this app needed to scale to support more views, nested paths, or breadcrumbs, I would have used **React Router**. Furthermore, if the project grew into a production-grade application requiring SEO optimization, fast initial loads, and Server-Side Rendering (SSR), I would have used **Next.js** instead of standard React with Vite.

Responsive with Tailwind: I used Tailwind's native media queries (md:, lg:, xl:) to hide the sidebar on mobile and show a bottom navigation bar, adapting the UI smoothly.

Search Bar with Debounce: In SearchBar.tsx, to prevent the geocoding API from being spammed with calls on every keystroke, I implemented a time control with a 400ms timeout before fetching.

Pagination: In DailyForecast.tsx, knowing that the API provides 30 days of data, I divided the results using React states into 6 pages of 5 days each, adding previous page, next page, and direct selectors from 1 to 6.

Flag Integration (FlagCDN): I decided against using third-party image APIs (like Unsplash) to avoid the complexity of setting environment variables in static deployments. Instead, I used the `country_code` parameter from the geocoding API to render country flags in the favorites list instantly and efficiently.

# Challenges

What was the most difficult part of the challenge?

How did you solve it?

Honestly, it was keeping track of all the required features and figuring out how to map them clean, modular, and reactive to the visual hierarchy in the reference image.

I approached it by stepping back from the code first. I drafted a checklist of all the features and structured how they would interact under the hood. Only then did I translate those logic blocks into prompt instructions for the AI. This systematic approach allowed me to successfully implement a robust weather icon mapping system, a persistent favorites list, smooth 30-day pagination, and dynamic view toggling.

# Trade-offs

If you had to simplify or postpone something, what was it?

Why did you make that decision?

Since the MVP only required switching between two core viewports ("Home" and "Favorites"), integrating a full routing library like React Router would have been over-engineering. I chose to use a simple reactive state to switch views, keeping the bundle size down.

Also, I initially wanted to fetch dynamic city background photos using a public image API. However, doing so would require setting up API keys, exposing them in a static client-side deployment, and running into strict rate limits. To keep the deployment lightweight, secure, and zero-config, I used a keyless, static Flag CDN to dynamically render national flags next to favorite cities instead.

# Improvements

Imagine you had another sprint.

What would you improve?

- Cº Fº toggle: a global unit context to allow users to toggle temperatures between Celsius and Fahrenheit.

- Production Architecture (Next.js): If we were scaling this application with more routes, user accounts, and configuration panels, I would migrate the Vite SPA architecture to a server-rendered Next.js framework to enable robust SEO, fast initial loads, and Server-Side Rendering (SSR).

- Design System & Semantic SEO: I would establish a formal design system (centralized color palette, typographic hierarchy, and theme tokens) and audits for semantic heading structures to improve SEO and screen-reader accessibility.

- Skeleton Screens & Caching: I would replace generic loading indicators with matching Skeleton Screens (skeleton loading placeholders) that occupy the exact layout footprint of the incoming data, preventing Cumulative Layout Shift (CLS) during API fetches.

# Self Assessment

Looking back at your implementation...

### What part are you most proud of?

I am incredibly proud of the mobile-first, responsive grid layout built using Tailwind. The UI transitions beautifully from desktop views down to narrow mobile viewports, achieving a highly polished and scalable MVP footprint with minimal code.

---

### What part would you refactor?

The custom view-state navigation. While a simple state-based toggler is perfectly suited for an MVP of this scale, I would refactor this to use a standard declarative router once a third view or parameterized URLs (like /city/:name) are introduced.

---

### If you had to start again, what would you do differently?

I would define my core TypeScript types and API response interfaces much earlier in the cycle. Nailing down the exact structure of the incoming weather payloads first would have made integrating the sub-components and setting up states significantly faster and less prone to minor runtime type mismatches.
